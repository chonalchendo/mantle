product-design.md, system-design.md

## Why

Skill compilation loads all 33 vault skills from the project-wide `skills_required` in state.md, regardless of which issue is active. This means implementation agents get every skill in context when they only need 3-5 relevant ones. Conventions and anti-patterns from the right skills get diluted by irrelevant content.

Separately, story implementation prompts generated by `/mantle:build` are task-focused but don't tell agents to read specific skills before implementing. During issue 35, three review issues (hardcoded data, CLI ergonomics, connector pattern) were all covered by loaded skills but ignored because the agent wasn't directed to prioritise them.

Both problems share a root cause: the build pipeline doesn't scope skills to the current issue.

## What to build

Two changes to the build pipeline:

### 1. Issue-scoped skill compilation

`compile_skills()` accepts an optional issue number. When provided, it compiles only the skills relevant to that issue (derived from issue content, story content, or an explicit `skills_required` field in the issue frontmatter) instead of the full project-wide list.

### 2. Skill references in story prompts

The build pipeline's implement step injects a `## Required reading` section into story prompts, listing the skill names relevant to that story. This directs agents to read specific skills before implementing, rather than hoping they notice the right ones in context.

### Flow

1. User runs `/mantle:build` for an issue
2. Build pipeline determines which skills are relevant to the current issue
3. `compile_skills()` compiles only those skills to `.claude/skills/`
4. Story prompts are generated with a `## Required reading` section listing relevant skill names
5. Implementation agents read the listed skills before coding, applying conventions and anti-patterns
