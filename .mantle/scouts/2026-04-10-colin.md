# Scout Report: colin

**Repository**: https://github.com/PrefectHQ/colin
**Analyzed**: 2026-04-10
**Project context**: AI workflow engine providing persistent memory, idea validation, and structured product development for Claude Code agents.

---

## Executive Summary

Colin is PrefectHQ's context compilation engine for AI agents — the project Mantle explicitly credits as inspiration for its "compiled context" pattern. It is a sophisticated Jinja2-based template engine with provider-backed data fetching, dependency tracking via Refs, manifest-driven incremental compilation, and LLM call caching. Overall signal quality is **high** — Colin solves the compilation half of what Mantle does, and its architecture reveals patterns we should adopt for our compiler, manifest, and provider layers. The single most important takeaway is Colin's Ref/Manifest system for tracking exactly what influenced each compiled output and enabling smart incremental recompilation.

---

## Findings by Dimension

### Architecture

- **Pattern name**: Unidirectional Layer Dependencies
  **What they do**: Core modules (compiler/, providers/, renders/) never import from higher-layer modules (api/, cli/). The api/ layer imports from core; cli/ imports from api/ only.
  **Why it's relevant**: Matches our core/ never imports cli/ rule but adds an intermediate api/ layer for programmatic use.
  **Recommendation**: Adopt — Enforce via import checks in CI; prevents architectural creep as the system grows.

- **Pattern name**: Protocol-Based Extension Architecture
  **What they do**: Core extension points defined as Protocols (InputPlugin, OutputPlugin, RefResolver) enabling dependency injection without inheritance.
  **Why it's relevant**: Allows pluggable providers and renderers without concrete base classes; cleaner testing.
  **Recommendation**: Adopt — Define extension protocols for vault backends, knowledge sources, and output formats.

- **Pattern name**: Two-Pass Compilation with Dependency Extraction
  **What they do**: Pass 1 parses all Jinja ASTs without rendering to extract ref() calls and build the dependency graph. Pass 2 does topological sort and renders in order.
  **Why it's relevant**: Catches dependency errors early, enables parallel compilation, prevents cyclic dependency surprises.
  **Recommendation**: Adapt — Valuable if our compilation grows complex; current single-pass may suffice for now.

- **Pattern name**: Manifest-Driven Staleness Model
  **What they do**: JSON manifest tracks output paths, ref versions, LLM calls, file hashes, and cache policies. Changes trigger recompilation only for affected documents.
  **Why it's relevant**: We already have a manifest (core/manifest.py) but Colin's is richer — tracks ref versions, not just content hashes.
  **Recommendation**: Adopt — Extend our manifest to track ref versions and LLM call metadata for smarter incremental builds.

- **Pattern name**: Compiler as Independent Library
  **What they do**: TemplateRenderer is usable independently of CompileEngine — rendering decoupled from orchestration/discovery.
  **Why it's relevant**: Our compiler.py mixes rendering and orchestration; separating them enables embedding in other contexts.
  **Recommendation**: Adopt — Extract template rendering from compilation orchestration for reusability.

- **Pattern name**: State Tree for Progress Tracking
  **What they do**: CompilationState and OperationState form a tree where each document has state (pending/cached/compiled/failed) with child operations.
  **Why it's relevant**: Enables rich progress UI and detailed error reporting for long-running operations.
  **Recommendation**: Adopt — Build a state tree for compilation and implementation loops.

### Coding Patterns

- **Pattern name**: Contextual Exception Classes
  **What they do**: Exceptions carry structured context (cycle path, target ref, failed dependency) as instance attributes, not just message strings.
  **Why it's relevant**: Our exceptions are basic; rich context enables better CLI error rendering and programmatic recovery.
  **Recommendation**: Adopt — Add structured context to our custom exceptions for better diagnostics.

- **Pattern name**: Hierarchical Pydantic Config Models
  **What they do**: Multi-level Pydantic models compose validation rules with field_validator (mode="before") for normalization and model_validator for cross-field checks.
  **Why it's relevant**: Cleaner than procedural validation; Pydantic validators handle complex config rules declaratively.
  **Recommendation**: Adopt — Use Pydantic validators with mode="before" for input normalization in our config/state models.

- **Pattern name**: ContextVar-Based State Propagation
  **What they do**: Uses contextvars.ContextVar to store current compilation context across async boundaries without threading locks.
  **Why it's relevant**: Avoids plumbing context through every function signature; propagates state cleanly through async Task trees.
  **Recommendation**: Adapt — Consider for our compile context if async compilation becomes needed; current sync approach is simpler.

- **Pattern name**: Batch Error Aggregation
  **What they do**: MultipleCompilationErrors stores dict[document_uri, list[Exception]] and skipped_uris, enabling complete error reports without stopping early.
  **Why it's relevant**: When compiling multiple skills/templates, we should report all failures, not just the first.
  **Recommendation**: Adopt — For batch operations (compile all skills, validate all issues), collect and report all errors together.

- **Pattern name**: Overload Declarations for Polymorphic APIs
  **What they do**: Uses @overload decorators to clarify multiple valid input/output type combinations for the same function.
  **Why it's relevant**: Helps type checkers and IDEs understand polymorphic APIs without runtime overhead.
  **Recommendation**: Adopt — Use typing.overload for functions with multiple valid signatures.

- **Pattern name**: Lazy-Loaded Plugin Registry
  **What they do**: Optional providers registered as strings (module paths), imported only on first use to keep base package lightweight.
  **Why it's relevant**: Keeps core package fast to import; users only pay for providers they actually use.
  **Recommendation**: Adopt — Use lazy imports for optional integrations (Obsidian CLI, GitHub API).

### Testing

- **Pattern name**: Collocated Test Project Fixtures
  **What they do**: CLI tests discover collocated project/ directories next to test files as self-contained fixtures with realistic file structures.
  **Why it's relevant**: Eliminates complex fixture setup; each test module has its own realistic project structure.
  **Recommendation**: Adopt — For CLI integration tests, place fixture projects alongside test files for self-contained E2E testing.

- **Pattern name**: Mock Agent Boundary Fixture
  **What they do**: A mock_agent fixture returns predictable output, allowing end-to-end flow testing without actual API calls.
  **Why it's relevant**: Aligns with our no-LLM-calls-in-tests rule while enabling integration testing of LLM-dependent workflows.
  **Recommendation**: Adopt — Create a similar fixture for any future LLM-integrated compilation or analysis features.

- **Pattern name**: Error Message Validation with match=
  **What they do**: pytest.raises(Error, match="pattern") validates error messages contain actionable context, not just that the right type was raised.
  **Why it's relevant**: Forces developers to write informative error messages; prevents tests passing with unhelpful errors.
  **Recommendation**: Adopt — Use match= in all pytest.raises calls to validate error message quality.

- **Pattern name**: Staleness Detection Testing
  **What they do**: Tests compile, modify source, compile again to verify only changed documents recompile and cached documents are preserved.
  **Why it's relevant**: Our compile doesn't clean up orphaned files (issue 49 learning); testing staleness explicitly would catch this class of bugs.
  **Recommendation**: Adopt — Write tests that exercise the compile, modify, recompile cycle for our manifest and skill compilation.

- **Pattern name**: Class-Based Test Organization
  **What they do**: Related tests grouped in classes (TestCompileEngine, TestOutputConfig) with descriptive method names focused on behavior.
  **Why it's relevant**: Makes test intent clearer; classes can share fixtures via setup methods.
  **Recommendation**: Adapt — Use classes for test modules with many related scenarios; keep flat functions for simple test files.

### CLI Design

- **Pattern name**: Nested Subcommand Groups
  **What they do**: Commands organized as colin <group> <verb> (e.g., colin mcp add, colin mcp auth clear) with semantic grouping.
  **Why it's relevant**: Directly addresses issue 48 (group mantle CLI commands in --help). Colin's Cyclopts-based grouping creates natural hierarchy.
  **Recommendation**: Adopt — Use Cyclopts command groups to organize mantle commands by workflow phase.

- **Pattern name**: Segregated Output Streams with Rich Markup
  **What they do**: Maintains separate Console() and Console(stderr=True) instances. Errors go to stderr with styled markup.
  **Why it's relevant**: Enables clean piping; scripts can capture stdout separately from errors.
  **Recommendation**: Adopt — Create shared CLI console instances with consistent error formatting.

- **Pattern name**: Contextual Error Messages with Recovery Suggestions
  **What they do**: Error messages include a recovery path: "Run 'colin init' to create a new project", "Use -f to confirm".
  **Why it's relevant**: Reduces friction for users. Every error answers "what now?" not just "what went wrong?"
  **Recommendation**: Adopt — Every CLI error should include a suggested next step.

- **Pattern name**: Rich Tables for Structured Output
  **What they do**: Uses rich.table.Table for list outputs with styled columns, alignment, and wrapping.
  **Why it's relevant**: More scannable than plain text for listing issues, stories, skills.
  **Recommendation**: Adopt — Use Rich tables for list/status outputs instead of plain text.

- **Pattern name**: Progress Spinners with State Trees
  **What they do**: Uses rich.live.Live with custom render_state() to show compilation progress as a document/operation tree.
  **Why it's relevant**: Long-running operations (compile, implement) feel responsive; users see real-time progress.
  **Recommendation**: Adapt — Implement progress trees for multi-story implementation and multi-skill compilation.

- **Pattern name**: Non-Interactive Fallback with Force Flag
  **What they do**: When stdin is not a TTY, fails with error instructing user to add -f/-y flag instead of hanging.
  **Why it's relevant**: Makes commands CI-friendly; prevents interactive prompts from blocking automation.
  **Recommendation**: Adopt — Any command requiring confirmation should check isatty() and support --force.

### Domain-Specific Features

- **Pattern name**: Refs as Replay Instructions
  **What they do**: Every external dependency captured in a Ref (provider, connection, method, args) that can be replayed to check staleness without re-fetching content.
  **Why it's relevant**: This is Colin's core insight — tracking exactly what influenced each compiled output. Directly relevant to our compilation pipeline.
  **Recommendation**: Adopt — Use Refs to track what knowledge sources feed each compiled skill, enabling "what changed?" queries.

- **Pattern name**: Two-Layer Storage (.colin/ vs output/)
  **What they do**: Build cache lives in .colin/compiled/ (git-versioned); published files live in output/. Private files (prefixed with _) compile but don't publish.
  **Why it's relevant**: We need to distinguish .mantle/ working state from published artifacts (skills, reports).
  **Recommendation**: Adapt — Consider underscore prefix convention for internal-only compiled artifacts.

- **Pattern name**: Human-in-the-Loop via output() Function
  **What they do**: Templates read their previously published file via output(), merge with new AI-generated content, preserving human edits across recompilation.
  **Why it's relevant**: Solves the technical problem of preserving human edits across compilation runs — key to our "bidirectional" differentiator.
  **Recommendation**: Adopt — Implement output() equivalent so human-edited skills/decisions survive recompilation.

- **Pattern name**: LLM Call Caching with Position-Based Lookup
  **What they do**: LLM calls cached by input hash + position ID. Previous output passed in context for stability. Unchanged inputs reuse cached output.
  **Why it's relevant**: Prevents redundant LLM calls during recompilation; enables iterative refinement without full regeneration.
  **Recommendation**: Adopt — Cache agent outputs and pass previous output in prompts for incremental improvement.

- **Pattern name**: Pluggable Data Source Providers
  **What they do**: Data fetching abstracted via providers (project://, mcp.*://, s3://). New sources added as new provider classes with lazy loading.
  **Why it's relevant**: We need to read from Obsidian vaults, GitHub issues, and potentially other knowledge sources.
  **Recommendation**: Adapt — Create providers for our knowledge sources when we need multi-source compilation.

- **Pattern name**: Sections as Addressable Content Units
  **What they do**: Templates define named sections; other documents reference specific sections via ref('other.md').sections.name.
  **Why it's relevant**: Enables fine-grained cross-referencing between decision logs, session summaries, and learnings.
  **Recommendation**: Adapt — Use for linking specific rationales across decisions and learnings when our vault grows large enough.

## Actionable Recommendations

### Adopt
> High confidence — bring this in directly with minimal adaptation.

- **Contextual Exception Classes** (from Coding Patterns): Add structured context (target, cycle path, related entities) as instance attributes to our custom exceptions for better CLI error rendering.
- **Error Message Validation with match=** (from Testing): Use match= in all pytest.raises calls to validate error messages contain actionable context.
- **Nested Subcommand Groups** (from CLI Design): Use Cyclopts command groups to organize mantle commands by workflow phase — directly addresses issue 48.
- **Contextual Error Messages with Recovery Suggestions** (from CLI Design): Every CLI error should include a suggested next step (e.g., "Run mantle init to create a project").
- **Rich Tables for Structured Output** (from CLI Design): Use Rich tables for list/status outputs.
- **Staleness Detection Testing** (from Testing): Write tests that exercise compile, modify, recompile cycles to catch orphaned-file bugs.
- **Batch Error Aggregation** (from Coding Patterns): For batch operations, collect all errors before reporting rather than failing on first.

### Adapt
> Worth doing but needs adjustment for our context.

- **Manifest-Driven Staleness Model** (from Architecture): Extend our manifest to track ref versions alongside content hashes — but scope to our current Jinja2 compilation, not Colin's full provider system.
- **Protocol-Based Extension Architecture** (from Architecture): Define Protocols for vault backends when we need pluggability — not needed until multi-backend support is considered.
- **Human-in-the-Loop via output()** (from Domain): Implement output reading in our Jinja2 templates to preserve human edits — scope to skills compilation initially.
- **Collocated Test Project Fixtures** (from Testing): Place realistic fixture projects alongside CLI test files — start with one or two key test modules.
- **Progress Spinners with State Trees** (from CLI Design): Add Rich progress for multi-story implementation — but only for operations that take >2 seconds.

### Skip
> Not relevant or not worth the cost for us right now.

- **Two-Pass Compilation** (from Architecture): Our compilation pipeline is simple enough for single-pass; two-pass adds complexity we don't need yet.
- **ContextVar-Based State Propagation** (from Coding Patterns): Our core is synchronous; contextvars add complexity without benefit until we go async.
- **Calendar-Aligned Staleness** (from Domain): No periodic report generation needed in current scope.
- **Refs as Replay Instructions** (from Domain): Full provider/ref system is overengineered for our current compilation needs — revisit if we add multi-source data fetching.
- **LLM Call Caching** (from Domain): Our compilation doesn't involve LLM calls; skip until we add AI-generated skill content.
