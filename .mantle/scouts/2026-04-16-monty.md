# Scout Report: monty

**Repository**: https://github.com/pydantic/monty.git
**Analyzed**: 2026-04-16
**Project context**: Mantle — AI workflow engine giving Claude Code persistent memory and structured product development.

---

## Executive Summary

Monty is a minimal, secure Python interpreter written in Rust, built to execute LLM-generated code safely with microsecond startup, state snapshotting, and sandboxed host interop. It is Pydantic AI's code-mode runtime — a sibling tool to Mantle, not a peer. Most domain-level findings from the agents (snapshot VM, polyglot Rust distribution, plugin registry) are a poor fit for Mantle's architecture and are classified as Skip. The genuine signal is in **Monty's Python surface layer**: typed exception hierarchy with a `.display()` method, semantic ANSI colour constants, fail-fast input validation, and test tooling (`inline_snapshot`, `dirty-equals`, dual-runner parametrisation). These patterns directly inform issues 51 (contextual CLI errors) and 42 (report-to-GitHub).

Signal quality: **medium** overall, **high** for CLI/error-handling patterns.

---

## Findings by Dimension

### Architecture

- **Suspension points + state serialization (dump/load)** — Monty pauses at external calls/OS ops/name lookups/futures, serialises VM state, resumes on host-provided result. **Skip.** Mantle's state is markdown files in git; git already supplies the durable checkpoint + replay layer. Importing a VM-level snapshot model inverts the design.
- **AbstractOS with NOT_HANDLED sentinel** — abstract base class for host-side FS ops; callers override dispatch or return NOT_HANDLED to delegate. **Skip.** Issue 56's hook seam is explicitly shell-scripts + exported env vars, not a Python plugin registry. Different abstraction level; adopting AbstractOS would violate the "tool-agnostic, no tracker libraries" constraint in the shaped design.
- **Thin PyO3 binding layer (Rust core + typed Python stubs)** — small typed Python surface, all logic in Rust. **Skip.** Mantle is pure Python by design; no identified perf bottleneck justifies a native-code rewrite.
- **Single-install polyglot distribution (PyPI + npm + crates.io)** — one Rust binary, per-language wrappers, single `pip install`. **Skip.** Mantle already ships as a single `mantle-ai` wheel; polyglot bindings are out-of-scope per product design.

### Coding Patterns

- **Typed exception hierarchy with `.display(format)` accessor** — `MontyError` base → `MontySyntaxError`, `MontyRuntimeError`, `MontyTypingError`; each holds the inner exception via `.exception()` and renders via `.display('type-msg')` / `.display('traceback')`. **Adopt.** Direct template for issue 51: `MantleError` base + specific subclasses (`ProjectNotFoundError`, `IssueNotFoundError`, `StateInvalidError`), each carrying a `.recovery_hint` attribute the CLI entry point can render consistently.
- **ANSI / Rich semantic colour constants** — `BOLD_RED`, `BOLD_CYAN`, `DIM`, `RESET` defined once, used everywhere; errors in red, actions/recovery hints in cyan, metadata dim. **Adopt.** Centralise Mantle's Rich styles in `src/mantle/cli/styles.py` (`ERROR`, `HINT`, `SUCCESS`) so issue 51's recovery-hint formatting is consistent across every error path without ad-hoc colouring.
- **Fail-fast input validation before execution** — Monty's CLI parses and validates all `-m` mount specs up front, emitting a single `mount {arg}: {reason}` error before any work starts. **Adopt.** Issue 42 (`/mantle:report`): validate `gh` availability + repo context + auth before prompting the user for issue details, so failures happen instantly, not after 3 input prompts.
- **Assertions for internal invariants, exceptions for boundaries** — `assert root_dir.is_absolute()` inside `OSAccess.__init__`; user-visible failures are always raised exceptions. **Adopt.** Reinforces the "fail-loudly" design from issue 57's learning: use assertions for contract violations in `core/`, raise typed exceptions at CLI entry points, never swallow either.
- **`match` with `case _: raise NotImplementedError` for dispatch** — exhaustive pattern matching over enum-like string commands. **Adapt.** Python 3.14 supports this. Mantle's CLI already uses Cyclopts dispatch, but internal dispatchers (e.g. state machine transitions, slice kind resolution) could gain the same exhaustiveness guarantee.
- **`TypedDict(total=False)` for optional-key config bags** — `ResourceLimits(TypedDict, total=False)` with documented fields per key. **Adapt.** Useful for Mantle's config passthrough (e.g. the `hooks.env:` dict in issue 56) where fields are opaque and extensibility matters.
- **`typing_extensions.deprecated` on legacy APIs** — legacy `run_monty_async` / `run_repl_async` decorated with the standard deprecation marker. **Adapt.** As Mantle's CLI surface grows, prefer the standard decorator over ad-hoc `DeprecationWarning` calls.

### Testing

- **`inline_snapshot` for API-contract + error-message tests** — golden-snapshot values embedded in the test source, auto-updated via pytest plugin. **Adopt.** High value for Mantle's CLI-output tests and compiled-command tests (where today we hand-write expected strings that drift). Pairs well with the existing `tests/test_staleness_regressions.py` style.
- **`dirty-equals` for order-independent collection comparisons** — `IsList(a, b, check_order=False)`, `IsPartialDict(...)`. **Adopt.** Removes the custom sort-then-compare boilerplate in manifest / issue / tag tests.
- **Parametrised dual-runner (MontyRunner vs CPythonRunner) for parity** — every behaviour test runs against both the real and the mocked implementation. **Skip.** Mantle's equivalent — real Obsidian CLI vs filesystem fallback — is already covered by `core/vault.py`'s subprocess boundary and the fallback tests; a second runner would duplicate, not extend, coverage.
- **Fixture library named by scenario** — fixtures like `test_dir`, `mount_readonly`, `mount_overlay` describe intent, not shape. **Adapt.** Current Mantle tests pass `tmp_path` around; named fixtures (`vault_with_issues`, `vault_after_init`) would make regression tests read as specifications.

### CLI Design

- **Two-part error rendering: severity prefix + actionable body** — Monty's CLI prints `error: <message>` with colour-coded severity, then contextual body indented on the next line. **Adopt.** Template for issue 51's output contract: line 1 `[red bold]Error:[/] what went wrong`, line 2 `[dim cyan]→ Run 'mantle init' to create a project`. Keeps parseability for automation while being human-readable.
- **Colour as semantic role, not decoration** — cyan is reserved for "action the user can take", red for "this failed". **Adopt.** Codifies the recovery-hint contract visually: every error's second line is cyan, every success is green; no ambiguity about which text is actionable.
- **Formatted duration output per run (`[time] ❯ [result]`)** — auto-scaled units, fixed significant digits. **Skip.** Mantle's Rich panels/progress bars already handle long-running feedback; duration strings would be a step backward visually.
- **REPL continuation prompts (`❯` vs `…`)** — state machine over incomplete-block input. **Skip.** Mantle has no interactive REPL and none is planned.

### Domain-Specific Features

- **Pydantic AI code-mode as the consumer** — Monty exists because Pydantic AI needs to execute agent-written Python safely. **Skip (as capability).** Mantle doesn't execute agent-written code; it dispatches Claude Code Agent subagents via prompts. However, *note the design partnership*: Monty is an execution runtime that Mantle-like orchestrators could plug into if Mantle ever needed to run tool-calling agents inside a sandbox (not on the near horizon).
- **Resource limits as loud-fail guardrails (memory, allocations, duration, recursion)** — configurable per-run, checked at every VM op, no unwind on violation. **Skip.** Mantle runs inside Claude Code's own sandbox; the orchestration layer doesn't need VM-level budgeting. If token-budget tracking becomes a need, it belongs in the prompt/telemetry layer, not a VM analogue.
- **Mount table with longest-prefix-first path resolution** — virtual FS with access modes (ReadOnly, ReadWrite, OverlayMemory), symlink validation, `..` escape rejection. **Skip.** Mantle's access control is git (project `.mantle/`) + filesystem (personal vault); no agent-facing virtual FS is needed.

## Actionable Recommendations

### Adopt
> High confidence — bring this in directly with minimal adaptation.

- **Typed exception hierarchy with `.recovery_hint`** (architecture / patterns): build the shared error utility for issue 51 around `MantleError` subclasses instead of free-form strings, so each raise site declares both the problem and the recovery in one object.
- **Semantic Rich colour constants** (CLI): centralise `ERROR`, `HINT`, `SUCCESS`, `DIM` styles in `src/mantle/cli/styles.py` before issue 51 migration, so every error path reuses the same tokens.
- **Two-line error output contract** (CLI): `Error: <message>` + `→ <recovery>`; defines the user-visible shape of issue 51.
- **Fail-fast input validation** (CLI): for issue 42, check `gh` availability, repo context, and auth before prompting for issue fields; apply the same "validate all, then act" discipline to other multi-step commands.
- **`inline_snapshot` for CLI / compiled-command tests** (testing): reduces churn in hand-written expected-output strings and matches the staleness regression style already in use.
- **`dirty-equals` for order-independent comparisons** (testing): retires the custom sort-compare helpers in manifest and tag tests.
- **Assertions for internal invariants** (patterns): reinforces the fail-loudly principle; pairs with typed exceptions at the CLI boundary.

### Adapt
> Worth doing but needs adjustment for our context.

- **Named scenario fixtures** (testing): migrate `tmp_path` callsites to descriptive fixtures (`vault_with_issues`, `vault_after_init`) as a gradual refactor — no big-bang rewrite.
- **`TypedDict(total=False)` for opaque config passthrough** (patterns): fits issue 56's `hooks.env:` passthrough exactly — document allowed keys in the TypedDict but accept arbitrary env vars.
- **`match` with exhaustive `case _`** (patterns): useful for internal dispatchers (state transitions, slice kinds); don't disturb Cyclopts routing.
- **`typing_extensions.deprecated`** (patterns): apply as CLI surface evolves, not proactively.

### Skip
> Not relevant or not worth the cost for us right now.

- **Snapshot VM / pause-resume** (architecture): git already provides Mantle's replay/resume; VM-level snapshots invert the design.
- **AbstractOS / plugin registry** (architecture): issue 56 is explicitly shell scripts, not Python plugins — adopting a class hierarchy would violate the shaped design.
- **Rust core + PyO3 bindings** (architecture): no perf justification; pure Python by design.
- **Polyglot npm/crates.io distribution** (architecture): out of scope per product design.
- **Dual-runner parametrisation** (testing): Mantle already covers real-vs-fallback at the subprocess boundary.
- **Resource limits / mount table** (domain): no agent-facing sandboxing needed — Claude Code is the sandbox.
- **REPL continuation prompts / formatted durations** (CLI): no REPL; Rich output supersedes.

## Backlog Implications

- **Issue 51 (contextual CLI errors with recovery) gains a concrete reference implementation.** The typed-exception-hierarchy + `.display()` + semantic colour constants pattern from Monty maps one-to-one onto the shared error utility this issue already prescribes. Shape-issue should evaluate "typed MantleError subclasses carrying `.recovery_hint`" as an approach, not free-form strings.
- **Issue 42 (`/mantle:report`) should bake in fail-fast validation.** Before capturing issue metadata, validate `gh` is installed, the user is authenticated, and the target repo is resolvable. Matches Monty's "validate all mount args before running" discipline and prevents wasted input.
- **Issue 56 (lifecycle hook seam) is not informed by Monty.** Monty's plugin/dispatch patterns operate at a fundamentally different level (in-process Python callables) than mantle's chosen seam (out-of-process shell scripts). Don't import that design.
- **Candidate new issue: test-tooling refresh.** Adopt `inline_snapshot` + `dirty-equals` across CLI/manifest tests. Low risk, reduces maintenance drag on tests that capture exact output or unordered collections. Could be a small slice (`cli`, `tests`) sized for one story.
