# Scout Report: caveman

**Repository**: https://github.com/JuliusBrussee/caveman
**Analyzed**: 2026-04-21
**Project context**: Mantle — AI workflow engine for Claude Code, currently focused on cost/context optimisation (active issue 84)

---

## Executive Summary

Caveman is a Claude Code plugin/skill whose entire product is cutting LLM tokens — output-side via a prompting style (~75% reduction across Lite/Full/Ultra/Wenyan intensity levels) and input-side via a `caveman-compress` tool (~46% reduction on prose memory files). Signal quality is **high** for Mantle right now because issue 84 (model tiering with measured baseline), issue 74 (command-token audit), and issue 79 (progressive-disclosure audit) are all variants of the same problem caveman has already productised. The single most important takeaway: Mantle is tuning *which model* runs each stage, but caveman shows that *compressing inputs/outputs* is an orthogonal, stackable lever Mantle currently has no mechanism for. Caveman's evals/ three-arm harness (baseline / terse-control / terse+skill) is also directly portable and fills issue 84's ac-04 measurement gap.

## Findings by Dimension

### Architecture

1. **Pattern: Single-source skill file with CI-driven multi-target sync**
   - What they do: Edit one canonical skills/caveman/SKILL.md; CI propagates copies with per-agent frontmatter to .cursor/, .windsurf/, .clinerules/, Claude Code plugin, etc.
   - Why it's relevant: Mantle currently ships Claude-Code-only (~/.claude/commands/mantle/). Issue 86 (plug external skill libraries) will eventually need a sync strategy.
   - Recommendation: **Skip (for now)** — Mantle's design principle #1 is Obsidian-native, not Obsidian-abstracted, and the CLI is Claude-Code-first. Multi-agent distribution is scope expansion, not optimisation.

2. **Pattern: Dual hooks — SessionStart (install ruleset) + UserPromptSubmit (per-turn reminder)**
   - What they do: SessionStart injects the full rule file once; UserPromptSubmit emits a one-line anchor (CAVEMAN MODE ACTIVE: full) on every turn so the model doesn't drift.
   - Why it's relevant: Mantle's long-lived commands (/mantle:build, /mantle:implement) spawn Agent subagents across many turns. Per-stage model/cost directives may drift mid-run (issue 84).
   - Recommendation: **Adapt** — Don't add hooks, but at each build.md stage boundary emit a one-line "Stage N/M: shape, model=sonnet" reminder so spawned agents stay anchored without reloading the full policy.

### Coding Patterns

3. **Pattern: Structural validation harness for lossy text transformations**
   - What they do: caveman-compress/scripts/validate.py extracts headings, code blocks, URLs, file paths, and bullet counts from both original and compressed files, then asserts they match (code/URLs byte-identical; bullets ±15%). Original is saved as .original.md before compression.
   - Why it's relevant: Mantle has no mechanism to prove a prompt rewrite didn't drop required placeholders, tool names, or acceptance-criteria structure. Issue 77's machine-verifiable acceptance criteria is a generalisation of this.
   - Recommendation: **Adopt** — Port the validator pattern to Mantle: any command that rewrites another command (future /mantle:compress for prompts) or renders Jinja2 output should validate structural invariants (frontmatter keys intact, CLI subcommand references intact, code fences intact) before commit.

4. **Pattern: File-type classifier (extension → content heuristics → keyword match)**
   - What they do: caveman-compress/scripts/detect.py classifies files as natural-language / code / config so only prose gets compressed. Handles extensionless files like CLAUDE.md.
   - Why it's relevant: Any Mantle-side compression would need to leave YAML frontmatter, code fences, and tool-call syntax untouched.
   - Recommendation: **Adapt** — Lift the classifier logic if/when /mantle:compress ships; don't reinvent.

### Testing

5. **Pattern: Three-arm contrastive eval harness (baseline / terse-control / terse+skill)**
   - What they do: Same prompts run under (1) no system prompt, (2) "Answer concisely.", (3) "Answer concisely.\n{SKILL.md}". The honest delta is (3) vs (2), isolating the skill's contribution from generic "be brief". Results committed as evals/snapshots/results.json; measure.py runs offline with tiktoken o200k_base — no API burn in CI.
   - Why it's relevant: Issue 84 ac-04 requires a before/after measurement saved to .mantle/telemetry/. A two-arm (Opus vs Sonnet) comparison would conflate "cheaper model" with "shorter response because mid-model." A control arm would isolate the tiering delta.
   - Recommendation: **Adopt** — Build .mantle/telemetry/ measurement as a three-arm harness from day one: (a) previous all-Opus default, (b) all-Sonnet control, (c) balanced preset from cost-policy. Commit results.json snapshots so cost regressions show up as PR diffs.

6. **Pattern: Approximate-but-deterministic token counting via tiktoken**
   - What they do: Use OpenAI's o200k_base tokenizer as a proxy for Claude's BPE. Report median/mean/min/max/stdev so noise is visible; document that ratios are meaningful, absolute numbers approximate.
   - Why it's relevant: Mantle wants CI-safe measurement without burning API budget on every PR.
   - Recommendation: **Adopt** — tiktoken is a clean way to get deterministic token counts for issue 84's baseline report.

### CLI Design

7. **Pattern: Output-format spec as an explicit template, not prose guidance**
   - What they do: caveman-review specifies output as L<line>: <severity> <problem>. <fix>. with explicit anti-patterns ("no 'I noticed'", "no restating"). caveman-commit is similarly rigid.
   - Why it's relevant: Issue 74 (audit commands for token-cost conciseness). Many Mantle slash-commands describe desired style in prose and rely on the model to interpret it loosely.
   - Recommendation: **Adopt** — For each Mantle command whose output is machine-consumed or user-scanned (/mantle:verify, /mantle:review, /mantle:status), add an explicit Output Format section with a one-line-per-item template and a short anti-pattern list. Easiest single change that measurably cuts command token spend.

8. **Pattern: Prompts that demonstrate the style they prescribe**
   - What they do: Caveman's own SKILL.md and command files are written in the terse style they ask the model to produce. Fragments, imperatives, tables.
   - Why it's relevant: Mantle command files (build.md, verify.md, implement.md) are often verbose prose and model verbosity by example.
   - Recommendation: **Adapt** — During the issue 74 audit pass, rewrite the iron-law sections and numbered steps of each /mantle:* command in imperative fragments. Target 20–30% per-command reduction without losing semantics.

### Domain-Specific Features

9. **Pattern: Intensity-level presets with runtime table filtering**
   - What they do: Six intensity levels (Lite / Full / Ultra / Wenyan variants) defined as rows in a single table in SKILL.md. Hook filters the table at runtime to inject only the active level's row. Presets parameterise behaviour; one source of truth.
   - Why it's relevant: Issue 84's cost-policy presets (budget/balanced/quality) are structurally identical. Caveman has proven the filter-at-runtime pattern is cheaper and easier to maintain than N separate prompt files.
   - Recommendation: **Adopt** — Implement cost-policy.md as a table with one row per preset and runtime selection, not three parallel prompt blocks. Extending with a fourth preset becomes adding a row.

10. **Pattern: Input compression of LLM-facing memory files (/caveman:compress)**
    - What they do: Dedicated skill that compresses CLAUDE.md and other prose context files, saving .original.md backups, validating structural invariants are preserved, cutting ~46% input tokens every session.
    - Why it's relevant: **Completely novel lever for Mantle.** Mantle currently tunes *which* model runs each stage (issue 84) but has no mechanism to compress what the model reads. .mantle/state.md, product-design.md, system-design.md, and CLAUDE.md are all read on every session start.
    - Recommendation: **Adapt** — File a new issue for /mantle:compress targeted at .mantle/ memory files + project CLAUDE.md. Natural follow-up to issue 84 model tiering; stacks with it multiplicatively. Keep the caveman .original.md backup pattern so compression is reversible.

11. **Pattern: Committed before/after token-cost snapshots as PR-reviewable diffs**
    - What they do: evals/snapshots/results.json committed to git. Any SKILL.md change shows up as a diff in the median/min/max token columns on PR.
    - Why it's relevant: Issue 84 ac-04 (baseline report) and the deferred A/B harness both benefit. Also gives issue 74 (command audit) a way to gate regressions.
    - Recommendation: **Adopt** — Commit .mantle/telemetry/baseline.json alongside the markdown baseline report so future model-tier or prompt changes are diff-reviewable.

12. **Pattern: Wenyan (classical Chinese) extreme-compression mode**
    - What they do: Treat a language with higher information density as just another intensity level.
    - Why it's relevant: Creative but orthogonal to Mantle's English-speaking user base.
    - Recommendation: **Skip** — Not in scope. Note the architectural lesson only: language-specific compression plugs into the same preset system, no new machinery needed.

## Actionable Recommendations

### Adopt
> High confidence — bring this in directly with minimal adaptation.

- **Three-arm eval harness** (Testing): Build issue 84's .mantle/telemetry/ baseline with a control arm so "model tier" and "generic brevity" effects don't conflate.
- **tiktoken-based token counting** (Testing): Deterministic, CI-safe measurement for issue 84's baseline report.
- **Output-format templates in commands** (CLI): Add explicit per-command Output Format sections targeting /mantle:verify, /mantle:review, /mantle:status. Biggest single win for issue 74.
- **Intensity-level table with runtime filtering** (Domain): Implement cost-policy.md as a single preset table, not parallel blocks.
- **Committed token-cost snapshots** (Domain): Make .mantle/telemetry/baseline.json a tracked file so regressions show up in PRs.
- **Structural validation harness** (Patterns): Port the headings/code/URL invariant checker for any future text-rewriting command.

### Adapt
> Worth doing but needs adjustment for our context.

- **Prompts demonstrate their own style** (CLI): During issue 74 audit, rewrite iron-law sections in imperative fragments. Adaptation: preserve Mantle's explicit-step structure; just tighten prose inside steps.
- **Per-stage reminder lines in build.md** (Architecture): No hooks needed — emit a one-line stage-transition reminder inside the prompt so spawned agents stay anchored to their tier.
- **Input compression for .mantle/ memory files** (Domain): File a new issue for /mantle:compress. Adapt caveman-compress's validator to Mantle's frontmatter schema (YAML keys intact, command references intact).

### Skip
> Not relevant or not worth the cost for us right now.

- **Multi-agent distribution (Cursor/Windsurf/Codex/etc.)** (Architecture): Mantle is Claude-Code-native by design principle. Scope expansion, not optimisation.
- **Flag-file state tracking with symlink-safety** (Architecture): Single-user local tool; overkill for Mantle's threat model.
- **Wenyan mode** (Domain): Out of scope for English-speaking audience.

## Backlog Implications

- **Issue 84 (active) directly benefits** from the three-arm eval harness, tiktoken-based measurement, and committed JSON snapshots. These are low-cost additions that make ac-04's baseline report significantly more rigorous than a single before/after row.
- **Issue 74 (audit commands for token-cost conciseness) gains a concrete method**: explicit output-format templates + rewriting prompts in the style they prescribe. Recommend scoping issue 74 around these two techniques rather than generic "make it shorter."
- **New issue candidate — /mantle:compress**: Caveman's input-side compression is a lever Mantle has no equivalent for, and it stacks with model tiering. Natural follow-up once issue 84 ships; would compress .mantle/state.md, product-design.md, system-design.md, project CLAUDE.md with a validator guarding structural invariants. Add to backlog.
- **Issue 79 (progressive-disclosure audit) gains a target**: the caveman pattern of "define behaviour once, filter at runtime by preset" is a good north star for how .mantle/ content should be loaded per-stage. Revisit 79 framing with this in mind.
