# Scout Report: rowboat

**Repository**: https://github.com/rowboatlabs/rowboat
**Analyzed**: 2026-04-10
**Project context**: AI workflow engine providing persistent context, idea validation, and structured product development for Claude Code

---

## Executive Summary

Rowboat is a multi-agent orchestration platform with a TypeScript/Python monorepo structure spanning CLI, web dashboard, and Electron app. Overall signal quality is **high** — Rowboat solves several problems Mantle is actively working on (skill injection, persistent memory, multi-agent orchestration) and its architecture demonstrates clean separation patterns that validate and extend Mantle's "core as library" principle. The single most important takeaway is their **skill catalog + agents-as-tools pattern**, which directly addresses Mantle's issue 52 (skill injection into planning) with a proven, production-grade approach.

---

## Findings by Dimension

### Architecture

- **Pattern name**: Interface-First Dependency Injection
  - **What they do**: All domain logic defined behind interfaces (IAgentsRepo, IRunsRepo, etc.) with concrete implementations registered in a DI container at startup.
  - **Why it's relevant**: Validates Mantle's "core as library" principle and shows how to formalize vault access as swappable persistence.
  - **Recommendation**: Adapt — Mantle's Python architecture could benefit from protocol classes in core/ defining repo interfaces, but full DI container (awilix-style) is over-engineering for a single-delivery-layer tool.

- **Pattern name**: Event-Sourced Run Logs
  - **What they do**: Agent runs are immutable append-only JSONL logs with typed events (MessageEvent, ToolInvocationEvent, etc.) broadcast via pub/sub bus.
  - **Why it's relevant**: Could enable queryable learnings (issue 41) — if story implementations were recorded as events, retrospectives and pattern detection become database queries rather than manual extraction.
  - **Recommendation**: Adapt — Event schemas are rigorous but JSONL is heavier than Mantle needs. The principle of typed, append-only records for decisions and implementations is worth adopting in a lighter form (structured YAML in .mantle/).

- **Pattern name**: Skills Catalog with Polymorphic Resolution
  - **What they do**: Skills are markdown documents with metadata, indexed in a catalog with alias resolution and on-demand loading via a builtin tool.
  - **Why it's relevant**: Directly addresses issue 52 (skill injection). Their catalog pattern makes skills discoverable and loadable without baking all context into prompts upfront.
  - **Recommendation**: Adopt — This validates the agent-driven skill selection approach already designed for issue 52.

- **Pattern name**: Repos as Storage Boundaries
  - **What they do**: Every persistence concern has an interface + filesystem implementation. Domain never touches file paths directly.
  - **Why it's relevant**: Mantle's vault.py partially does this but isn't formalized as a pattern across all artifact types (stories, learnings, shaped issues each have ad-hoc I/O).
  - **Recommendation**: Adapt — Formalizing repo interfaces would improve testability but shouldn't be done speculatively. Apply when adding new artifact types or refactoring existing ones.

- **Pattern name**: Monorepo with Nested Package Hierarchy
  - **What they do**: Two-level monorepo: apps/cli/, apps/rowboat/ (web), apps/x/ (Electron) sharing core packages.
  - **Why it's relevant**: Shows how to share core logic across delivery layers (CLI, web, desktop).
  - **Recommendation**: Skip — Mantle is a single-delivery tool (CLI). Monorepo adds complexity without benefit until a second delivery layer is built.

### Coding Patterns

- **Pattern name**: Pydantic Schema-First API Design
  - **What they do**: Every data structure crossing service boundaries is a Pydantic BaseModel with model_validate() and model_dump(). Uses Literal types for status fields and discriminated unions for message types.
  - **Why it's relevant**: Mantle already uses Pydantic but inconsistently across boundaries. Extending this pattern to all CLI/core boundaries would reduce runtime type errors.
  - **Recommendation**: Adopt — Use Literal types for all status enums and discriminated unions for polymorphic data. This is already aligned with Mantle conventions but could be applied more consistently.

- **Pattern name**: Decorator-Based Cross-Cutting Concerns
  - **What they do**: @require_signed_request decorator handles auth, JWT validation, and body hashing without polluting route handlers.
  - **Why it's relevant**: Mantle CLI commands could use decorators for config validation, error reporting, and prerequisite checking.
  - **Recommendation**: Adapt — Useful pattern but premature for Mantle's current scale. Worth remembering when CLI commands start sharing validation logic.

- **Pattern name**: Background Task Orchestration via asyncio.Semaphore
  - **What they do**: JobService uses asyncio.Semaphore(5) with background heartbeat tasks and event-based stop signals for bounded concurrent processing.
  - **Why it's relevant**: Clean async pattern for when Mantle implements long-running workflows or parallel operations.
  - **Recommendation**: Skip — Mantle doesn't have async workers yet. File this for future reference.

- **Pattern name**: Function Registry with Runtime Type Coercion
  - **What they do**: Maps function names to callables, validates parameters via inspect.signature(), and auto-converts types before dispatch.
  - **Why it's relevant**: Battle-tested safe dispatch pattern for tool execution from untrusted input.
  - **Recommendation**: Skip — Mantle delegates tool execution to Claude Code's native tools, not a custom dispatcher.

### Testing

- **Pattern name**: Dual-layer Testing (Unit + Integration)
  - **What they do**: Tests split between core function unit tests and HTTP endpoint integration tests, with identical error cases tested at both layers.
  - **Why it's relevant**: Mantle has a similar CLI/core split. Testing critical paths at both layers ensures errors are caught at the right place.
  - **Recommendation**: Adopt — For Mantle CLI commands that are more than thin routing, test both the core function and the CLI command separately. This is especially relevant for commands with complex argument parsing.

- **Pattern name**: Roundtrip Serialization Testing
  - **What they do**: Explicitly test double-JSON encoding and serialization edge cases at service boundaries.
  - **Why it's relevant**: Mantle has YAML parsing, frontmatter, and file I/O boundaries that similarly need roundtrip fidelity tests.
  - **Recommendation**: Adopt — Add explicit roundtrip tests for YAML frontmatter parsing and serialization, especially for edge cases (special characters, multiline strings, nested structures).

- **Pattern name**: Inline Test Rationale Comments
  - **What they do**: Every test includes a 1-line comment explaining what condition is being tested and why we expect that behavior.
  - **Why it's relevant**: Makes test intent explicit for future maintainers debugging failures.
  - **Recommendation**: Skip — Good practice but not a gap for Mantle. Test names and Google-style docstrings already serve this purpose.

### CLI Design

- **Pattern name**: Rich Terminal Output with Box-Drawing Characters
  - **What they do**: StreamRenderer uses Unicode box-drawing characters with ANSI colors to structure multi-line agent output into visually distinct sections (tool calls, reasoning, results).
  - **Why it's relevant**: Mantle's CLI output is plain text. Structured visual output would improve readability for complex operations like compile logs, implementation output, and status displays.
  - **Recommendation**: Adopt — Implement a similar output renderer for Mantle commands using rich (Python library) to improve readability of multi-step operations.

- **Pattern name**: Explicit Mutual Exclusivity Validation
  - **What they do**: import command uses .check() to validate that either --example or --file is provided but not both, with specific user-friendly error messages.
  - **Why it's relevant**: Aligns with issue 51 (contextual CLI errors). Explicit validation with recovery suggestions beats framework defaults.
  - **Recommendation**: Adopt — Implement in cyclopts commands with explicit validators that suggest the correct usage when flags conflict.

- **Pattern name**: Flat Task-Focused Subcommand Hierarchy
  - **What they do**: Commands are flat and single-purpose (import, export, list-examples, model-config, ui). No nested subcommands.
  - **Why it's relevant**: Mantle has some nesting (e.g., config subcommands). Flat structure is simpler to discover and document.
  - **Recommendation**: Adapt — Consider flattening deeply nested commands. The help panels approach (issue 48) may address discoverability without flattening, so evaluate both.

- **Pattern name**: Descriptive Command Introductions
  - **What they do**: Command descriptions hint at common usage patterns ("Import an example workflow (--example) or custom workflow from file (--file)").
  - **Why it's relevant**: Reduces help-seeking by embedding usage hints in descriptions.
  - **Recommendation**: Adopt — Enhance Mantle command descriptions to include usage hints and common patterns.

### Domain-Specific Features

- **Pattern name**: Declarative Skill Injection
  - **What they do**: Maintain a skill catalog where each skill is a markdown document with title, summary, and guidance content, dynamically assembled into agent instructions based on context.
  - **Why it's relevant**: Directly addresses issue 52 (skill injection into planning). Validates the agent-driven skill discovery approach over static injection.
  - **Recommendation**: Adopt — This is the pattern Mantle should follow for issue 52. Skills as cataloged, loadable documents with a discovery mechanism.

- **Pattern name**: Persistent Agent Memory Files (Sidecar Pattern)
  - **What they do**: Plain markdown memory files (user.md, preferences.md) automatically injected into every agent run, with a save-to-memory tool for updates during runs.
  - **Why it's relevant**: Validates Mantle's session logging and auto-briefing approach. The sidecar pattern (always-injected memory) is simpler than retrieval-augmented generation.
  - **Recommendation**: Skip — Mantle already has this via compiled resume.md and session logs. Validated, not novel.

- **Pattern name**: Agents-as-Tools Pattern
  - **What they do**: Agents declare other agents as tools in frontmatter. Spawning creates isolated subflows with state tracking and full audit trails.
  - **Why it's relevant**: Enables Mantle's "prompt orchestration with fresh context windows per story" pattern. The subflow tracking provides audit trails for retrospectives.
  - **Recommendation**: Adapt — The subflow event tracking is worth studying for Mantle's implementation loop. Currently implement.md spawns agents but doesn't track their execution as structured events.

- **Pattern name**: Pipeline-Based Knowledge Graph Construction
  - **What they do**: Sequential agent pipeline (labeling -> graph building -> tagging) with checkpoint state allowing mid-pipeline resumption.
  - **Why it's relevant**: Relevant to compiled context and staleness detection (issue 50). Incremental, checkpoint-based compilation avoids rebuilding from scratch.
  - **Recommendation**: Adapt — The checkpoint-and-resume pattern is valuable for long-running context builds. Apply to session compilation: build context incrementally with checkpoints.

## Actionable Recommendations

### Adopt
> High confidence — bring this in directly with minimal adaptation.

- **Declarative Skill Injection** (from Domain/Architecture): Validates and accelerates issue 52's design — skill catalog with on-demand loading via agent tools.
- **Rich Terminal Output** (from CLI Design): Use Python's rich library to add structured, colored output for multi-step operations. Immediate UX win.
- **Explicit Mutual Exclusivity Validation** (from CLI Design): Implement in cyclopts with user-friendly error messages and recovery suggestions. Feeds directly into issue 51.
- **Dual-layer Testing** (from Testing): Test critical paths at both CLI and core layers independently, especially commands with complex argument parsing.
- **Pydantic Literal/Union Types** (from Coding Patterns): Apply Literal types for status enums and discriminated unions more consistently across core models.

### Adapt
> Worth doing but needs adjustment for our context.

- **Event-Sourced Run Logs** (from Architecture): Lighter-weight structured records for implementation decisions and story outcomes — enables issue 41 (querying learnings for patterns). Use YAML in .mantle/ rather than JSONL.
- **Agents-as-Tools Subflow Tracking** (from Domain): Track agent execution during implementation as structured events for richer retrospectives. Don't adopt the full subflow state machine — Mantle's prompt-based orchestration is simpler and sufficient.
- **Checkpoint-and-Resume Pipeline** (from Domain): Apply to compilation — build context incrementally with checkpoints so resumption doesn't rebuild from scratch. Relevant to issue 50.
- **Repos as Storage Boundaries** (from Architecture): Formalize as protocol classes when refactoring vault access, not speculatively.

### Skip
> Not relevant or not worth the cost for us right now.

- **Monorepo Structure** (from Architecture): Mantle is CLI-only. Monorepo complexity not justified until a second delivery layer exists.
- **asyncio Semaphore Workers** (from Coding Patterns): No async workers in Mantle's roadmap. File for future reference.
- **Function Registry Dispatcher** (from Coding Patterns): Claude Code's native tools handle dispatch. Custom dispatcher unnecessary.
- **Persistent Agent Memory Sidecar** (from Domain): Mantle already has this via compiled resume.md and session logs. Validated, not novel.
- **Instruction Cache Invalidation** (from Domain): Implementation detail. Mantle should fill the content layer (what context to include) before optimizing caching.

## Backlog Implications

- **Issue 52 (Skill injection)**: Strongly reinforced. Rowboat's skill catalog with on-demand loading validates the agent-driven discovery approach. The catalog + alias resolution pattern provides a concrete implementation reference.
- **Issue 51 (Contextual CLI errors)**: Reinforced by Rowboat's explicit mutual exclusivity validation with recovery hints. Consider pairing with richer output formatting (rich library) for a cohesive CLI UX improvement.
- **Issue 41 (Query learnings for patterns)**: Rowboat's event-sourced run logs suggest that structured, typed records would make pattern detection far more effective than querying freeform markdown. Consider adapting the event schema concept when implementing this issue.
- **Issue 50 (Staleness detection)**: The checkpoint-and-resume pipeline pattern is relevant — incremental compilation with checkpoints would make staleness detection both faster and more testable.
